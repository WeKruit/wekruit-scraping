# 01-01 Plan: Config, Source Registry, Run Metadata, And Raw Staging Contract

## Objective
Define the phase-1 ingest contract so every official source run uses one config surface, one source registry, one run manifest shape, and one raw staging layout before source-specific fetch logic is implemented.

## Files To Modify
- Create: `researcher/researcher_config.py`
- Create: `researcher/researcher_sources.py`
- Create: `researcher/researcher_run_state.py`
- Create: `researcher/researcher_pipeline.py`
- Create: `researcher/tests/test_researcher_config.py`
- Create: `researcher/tests/test_researcher_run_state.py`
- Create: `researcher/tests/test_researcher_sources.py`
- Modify: `researcher/README.md`

## Tasks
1. Define a single phase-1 config module in `researcher/researcher_config.py`.
   - Include only fields needed for official ingest foundation: data root, OpenAlex credentials / polite-contact headers, Crossref mailto, default per-source page size, retry count, backoff, and run mode flags.
   - Keep the config flat and environment-driven; do not add settings for merge, enrichment, ranking, or crawling.
   - Expose one typed loader function that returns the full runtime config object for the pipeline.

2. Define the source registry in `researcher/researcher_sources.py`.
   - Register only `openalex` and `crossref`.
   - For each source, record: source name, source type, enabled flag, required config keys, supported operations, raw output paths, and checkpoint keys.
   - Make the registry the only place where source IDs are declared so later plans cannot invent ad hoc names.

3. Define run metadata and raw staging contracts in `researcher/researcher_run_state.py`.
   - Create one run manifest shape that records: `run_id`, `started_at`, `completed_at`, `status`, `preset_type`, `preset_value`, `since_year`, `max_records`, `sources_requested`, `sources_completed`, `config_fingerprint`, and per-source stats.
   - Create one source-attempt shape that records: request parameters, retry count, checkpoint cursor, page count, record count, and error summary.
   - Define raw staging directory helpers under `researcher/data/runs/<run_id>/...`.
   - Define append-only JSONL writers for raw paper / author / metadata records and JSON writers for run/checkpoint manifests.

4. Define pipeline orchestration skeleton in `researcher/researcher_pipeline.py`.
   - Parse CLI arguments for the phase-1 run envelope only: preset type, preset value, since year, max records, source selection, replay mode, and incremental mode.
   - Validate source selection against `researcher_sources.py`.
   - Create run directories and persist the initial manifest before any source call happens.
   - Do not implement source fetching here yet; wire stubs only for the future `openalex` and `crossref` steps.

5. Add focused tests for the contract.
   - `test_researcher_config.py`: config loader accepts required ingest env vars and rejects missing required source settings.
   - `test_researcher_sources.py`: registry exposes only `openalex` and `crossref` with stable IDs and required operations.
   - `test_researcher_run_state.py`: run initialization creates the expected directory tree and manifest payload; raw writer preserves append-only JSONL semantics.

6. Update `researcher/README.md` with phase-1 contract documentation.
   - Document the run directory layout.
   - Document the allowed phase-1 CLI modes.
   - Document that phase 1 is raw ingest only and excludes merge/enrichment/ranking.

## Done Criteria
- `researcher/researcher_config.py` loads one consistent config object for phase-1 ingest
- `researcher/researcher_sources.py` exposes a stable registry for `openalex` and `crossref` only
- `researcher/researcher_run_state.py` can create a run manifest and write raw JSONL artifacts into a run-scoped directory
- `researcher/researcher_pipeline.py` can start a run, validate CLI/config input, and write initial manifests without performing fetches yet
- Tests cover config validation, source registry stability, and run-state artifact creation
- `researcher/README.md` explains the phase-1 contract in repo terms

## Risks / Dependencies
- The config surface must stay phase-1-only; if later-source or merge settings leak in here, the executor is overbuilding
- The manifest must be source-native and operational, not canonical-domain shaped
- The raw layout must be fixed before `01-02`; changing it later will break replay semantics
